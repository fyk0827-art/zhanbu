# 支付对接文档（问答平台 ↔ 报告平台）

> **报告用户页（浏览器打开）**：[http://39.97.224.240:8842](http://39.97.224.240:8842) — Career & Wealth Report  
> **报告后端 API（服务端调用）**：`http://39.97.224.240:8848`  
> **验签密钥（WEBHOOK_SECRET）**：由报告平台提供，配置在 `qa-backend/application.yml` 的 `partner.webhook-secret`

 *验签密钥（"<YOUR_WEBHOOK_SECRET>"）*

---

## 重要说明：8842 与 8848 的区别

| 地址 | 用途 | 谁发起请求 | 是否 POST JSON 确认支付 |
|------|------|------------|-------------------------|
| `:8848` | 合作方**后端 API** | **你的服务器**（Java 后端） | ✅ 是，见下文 `confirm-payment` |
| `:8842` | 合作方**用户前端页面** | **用户浏览器**（整页跳转） | ❌ 否，不接收代理方的支付确认 JSON |

**结论**：支付成功确认必须请求 **`8848`**，不能向 **`8842`** 发送 `confirm-payment` 的 JSON。  
8842 仅用于打开返回的 `frontendUrl`，让用户填写出生信息并生成报告。

## 一、推荐完整流程

```
用户在问答平台答题 → 点击支付
        ↓
问答平台后端创建本地支付单（tradeNo）
        ↓
问答平台后端 POST 8848 /api/partner/confirm-payment（带 HMAC 签名）
        ↓
8848 返回 frontendUrl（指向 8842，带 orderId）
        ↓
浏览器跳转到 frontendUrl，例如：
http://39.97.224.240:8842/#/?orderId=BP17xxxx
        ↓
用户在 8842 页面填写姓名、生日、地点 → 生成报告（已预付费，无需再付）
```

## 二、支付成功确认接口（核心，请求 8848）

### 2.1 基本信息

| 项目 | 值 |
|------|-----|
| 方法 | `POST` |
| URL | `http://39.97.224.240:8848/api/partner/confirm-payment` |
| Content-Type | `application/json; charset=utf-8` |
| 鉴权 Header | `X-Partner-Signature`（HMAC-SHA256 十六进制小写） |

**必须由代理方服务端调用**，禁止在用户浏览器里直接调用（会暴露密钥）。

### 2.2 请求 JSON 格式

```json
{
  "tradeNo": "pay_1730000000000_a1b2c3d4",
  "amount": 1900,
  "payerContact": "user@example.com"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `tradeNo` | string | **是** | 代理方支付流水号，**幂等键**；同一 `tradeNo` 重复调用返回同一订单 |
| `amount` | number | 否 | 金额，单位：**分**。标准价 `1900` = $19.00；传入且非 0 时须与报告平台配置一致 |
| `payerContact` | string | 否 | 用户邮箱或手机号，便于对账 |

**最小请求示例（仅必填字段）：**

```json
{
  "tradeNo": "pay_1730000000000_a1b2c3d4"
}
```

> 本问答平台后端在 mock 模式下会固定发送 `amount: 1900`（见 `partner.standard-amount-cents`）。

### 2.3 签名算法

```text
rawBody     = 请求体的原始 JSON 字符串（与发送内容完全一致，勿多空格）
signature   = hex_lower( HMAC-SHA256( WEBHOOK_SECRET, rawBody ) )
Header      = X-Partner-Signature: <signature>
```

**示例（与上面 Body 一致）：**

```text
Body: {"tradeNo":"pay_1730000000000_a1b2c3d4","amount":1900}
Secret: <YOUR_WEBHOOK_SECRET>
→ 用 Secret 对 Body 做 HMAC-SHA256，输出 64 位小写十六进制字符串
```

### 2.4 成功响应 JSON 格式

**HTTP 状态码**：`200`

```json
{
  "ok": true,
  "alreadyConfirmed": false,
  "orderId": "BP1780xxxx",
  "prepaid": true,
  "reportPending": true,
  "frontendUrl": "http://39.97.224.240:8842/#/?orderId=BP1780xxxx",
  "amount": 1900,
  "amountDisplay": "19.00",
  "hint": "请引导用户打开 frontendUrl，在前端填写信息并生成报告",
  "tradeNo": "pay_1730000000000_a1b2c3d4"
}
```

| 字段 | 说明 |
|------|------|
| `ok` | 必须为 `true` 表示成功 |
| `frontendUrl` | **必须**引导用户浏览器打开的链接，域名一般为 **8842** |
| `orderId` | 报告平台订单号，已标记预付费 |
| `alreadyConfirmed` | `true` 表示该 `tradeNo` 此前已确认过（幂等） |
| `prepaid` | 是否已预付费 |
| `reportPending` | 是否待用户在前端生成报告 |

### 2.5 失败响应（常见）

| 情况 | 表现 |
|------|------|
| 签名错误 | HTTP 4xx，Body 含错误信息 |
| `amount` 与平台配置不一致 | 拒绝确认（例如传了 999 而标准价为 1900） |
| `tradeNo` 为空 | 参数校验失败 |

代理方应记录完整响应体与 HTTP 状态码，便于排查。

### 2.6 cURL 测试示例

```bash
SECRET="<YOUR_WEBHOOK_SECRET>"
TRADE_NO="pay_test_$(date +%Y%m%d%H%M%S)"
BODY="{\"tradeNo\":\"$TRADE_NO\",\"amount\":1900}"

SIG=$(echo -n "$BODY" | openssl dgst -sha256 -hmac "$SECRET" | awk '{print $2}')

curl -s -X POST "http://39.97.224.240:8848/api/partner/confirm-payment" \
  -H "Content-Type: application/json; charset=utf-8" \
  -H "X-Partner-Signature: $SIG" \
  -d "$BODY"
```

将响应中的 `frontendUrl` 复制到浏览器地址栏，即可进入 [8842 报告页](http://39.97.224.240:8842)。

### 2.7 PowerShell 测试示例（Windows）

```powershell
$Secret = "<YOUR_WEBHOOK_SECRET>"
$TradeNo = "pay_test_" + (Get-Date -Format "yyyyMMddHHmmss")
$Body = "{`"tradeNo`":`"$TradeNo`",`"amount`":1900}"

$hmac = New-Object System.Security.Cryptography.HMACSHA256
$hmac.Key = [Text.Encoding]::UTF8.GetBytes($Secret)
$hash = $hmac.ComputeHash([Text.Encoding]::UTF8.GetBytes($Body))
$Sig = -join ($hash | ForEach-Object { $_.ToString("x2") })

Invoke-RestMethod -Method Post `
  -Uri "http://39.97.224.240:8848/api/partner/confirm-payment" `
  -ContentType "application/json; charset=utf-8" `
  -Headers @{ "X-Partner-Signature" = $Sig } `
  -Body $Body
```

## 三、8842 用户页（非支付 API）

### 3.1 如何「访问」8842

支付确认成功后，**不要**向 8842 POST 支付 JSON。应使用：

```http
GET http://39.97.224.240:8842/#/?orderId=BP1780xxxx
```

由用户浏览器整页跳转（`window.location.href = frontendUrl`）或新窗口打开。

### 3.2 用户在 8842 上做什么

1. 页面提示「您已完成支付，请填写出生信息」
2. 选择报告类型，填写姓名、生日、出生地点等
3. 生成报告；前端内部会将报告与 `orderId` 绑定（`bind-report`），用户直接查看全文

**代理方无需为 8842 开发接口调用**，也无需向 8842 发送 `confirm-payment` 请求体。

### 3.3 URL 参数说明

| 参数 | 来源 | 说明 |
|------|------|------|
| `orderId` | `confirm-payment` 响应中的 `orderId` | 已预付费订单号，嵌入在 `frontendUrl` 的 hash 查询串中 |

示例：

```text
http://39.97.224.240:8842/#/?orderId=BP1780xxxx
```

## 四、本问答平台对外 API（可选了解）

前端不直接调 8848，而是调本地 Java 后端，由后端代签并调用报告平台。

### 4.1 创建支付单

**POST** `http://localhost:8080/api/payments/create`

请求：

```json
{
  "questionId": 1
}
```

响应：

```json
{
  "success": true,
  "data": {
    "tradeNo": "pay_1730000000000_a1b2c3d4",
    "amount": 9.99,
    "currency": "USD",
    "status": "pending"
  }
}
```

### 4.2 完成支付（mock：自动调 8848 并返回跳转地址）

**POST** `http://localhost:8080/api/payments/complete`

请求：

```json
{
  "tradeNo": "pay_1730000000000_a1b2c3d4"
}
```

响应：

```json
{
  "success": true,
  "data": {
    "tradeNo": "pay_1730000000000_a1b2c3d4",
    "status": "completed",
    "orderId": "BP1780xxxx",
    "frontendUrl": "http://39.97.224.240:8842/#/?orderId=BP1780xxxx"
  }
}
```

前端收到 `frontendUrl` 后立即：

```javascript
window.location.href = completed.frontendUrl;
```

## 七、FAQ

**Q：能不能直接请求 `http://39.97.224.240:8842` 完成支付？**  
A：不能。8842 是用户填写报告信息的 Web 页面；支付确认 JSON 必须 POST 到 **8848** 的 `confirm-payment`。

**Q：`frontendUrl` 一定是 8842 吗？**  
A：是。成功响应里的 `frontendUrl` 通常形如 `http://39.97.224.240:8842/#/?orderId=...`，以接口实际返回为准。

**Q：同一 `tradeNo` 重复调用会怎样？**  
A：幂等返回同一 `orderId` 与 `frontendUrl`，`alreadyConfirmed` 可能为 `true`。

**Q：支付成功但 8842 打不开？**  
A：先确认 8848 是否返回 200 和有效 `frontendUrl`；再检查本机网络、防火墙是否允许访问 `39.97.224.240:8842`。
